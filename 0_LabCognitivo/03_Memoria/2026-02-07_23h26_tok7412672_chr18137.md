## Assistente

Ótima pergunta! Isso é o cenário clássico de **colaboração no Git**. Vou explicar:

## O que acontece

### Cenário: Você e ela editam e fazem push

1. **Quem fizer push primeiro** — funciona normalmente, sem problemas.

2. **Quem fizer push segundo** — o Git **rejeita** o push com uma mensagem tipo:
   ```
   ! [rejected] main -> main (fetch first)
   error: failed to push some refs to '...'
   hint: Updates were rejected because the remote contains work that you do not have locally.
   ```

### Por quê?
O Git protege vocês. Ele não quer sobrescrever o trabalho de ninguém.

### Como resolver?
A segunda pessoa precisa fazer **pull antes do push**:

```
git pull origin main
```

Aí existem **dois cenários**:

---

**A) Editaram arquivos diferentes** (ou partes diferentes do mesmo arquivo)
- O Git faz o **merge automático** — junta tudo sozinho.
- Depois é só fazer `git push` normalmente.

---

**B) Editaram a mesma parte do mesmo arquivo**
- Acontece um **merge conflict** (conflito).
- O Git marca no arquivo as duas versões assim:
  ```
  <<<<<<< HEAD
  (sua versão)
  =======
  (versão dela)
  >>>>>>> origin/main
  ```
- Vocês escolhem qual versão manter (ou combinam as duas), salvam, e fazem commit + push.

---

## Resumo prático do fluxo a dois

```
git pull          ← sempre antes de começar a editar
(faz as edições)
git add .
git commit -m "mensagem"
git pull          ← de novo, antes de dar push (por segurança)
git push
```

A regra de ouro: **sempre dê pull antes de começar a trabalhar e antes de fazer push**. Isso minimiza conflitos.
